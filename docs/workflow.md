# 开发流程：md = 源代码

## 核心理念

- **md 是源代码，ts 是 md 的实现**。
- 每个模块在 `docs/specs/` 下有对应的 `.md` 规范文件。
- spec 描述契约：目的、接口、行为、不变量、错误情形。
- 代码实现 spec；测试验证实现符合 spec。

## 工作流（四步）

1. **写或更新 spec** — 在 `docs/specs/<module>.md` 写清楚：
   - 目的（为什么需要这个模块）
   - 接口（导出的类型与函数签名）
   - 行为（成功路径、边界、错误）
   - 不变量（永远成立的断言）
2. **写测试** — 测试是 spec 的可执行版本，逐条对应 spec 的行为条目。
3. **写实现** — 写最少的代码让测试通过，不要超出 spec 范围。
4. **同步 spec** — 如果实现过程中发现 spec 不准确，**先改 spec，再改代码**。

## spec 与代码不一致时

**以 spec 为准**。代码要么修正以匹配 spec，要么提出改 spec 的理由并更新 spec。
不允许"代码现在是这样所以 spec 也这样写"——这是倒因为果。

## spec 文件模板

每个 spec 至少包含以下段落（见 [specs/README.md](specs/README.md) 的完整模板）：

- **概述** — 这个模块做什么、为什么存在。
- **接口** — 导出的类型、函数签名。
- **行为** — 每个函数的成功路径、边界、错误。
- **示例** — 典型调用与返回。
- **不变量** — 永远成立的断言（例如："任务状态只能按 todo → in_progress → in_review → done 流转"）。

## PR 规则

- 动 spec 必须动实现与测试。
- 动实现必须确认对应 spec 仍然准确。
- 新功能：先在 PR 中加入或修改 spec，再加实现与测试。
- 纯重构：spec 不变，实现和测试同步调整。
