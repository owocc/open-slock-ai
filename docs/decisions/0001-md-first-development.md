# 0001. 采用 md-first 开发模式

- 状态：accepted
- 日期：2026-05-12

## 背景

作为对标 slock.ai 且支持高度定制与扩展的开源 AI Agent 协同平台，OpenSlock 面临着极高的模块解耦度和架构一致性要求。如果**规范散落在代码里**，会导致各模块在开源开发者与 Agent 协作开发中缺乏明确的协议规范（如 API breaking changes、Agent 自动开发的契约门控以及 schema 迁移路径冲突等）。

我们需要一个顶层约束：让设计规范（Specification）独立于具体实现，成为可审阅、可由机器验证的制品。

## 决策

采用**文档优先（md-first）**开发模式：

- 每个模块在 `docs/specs/<module>.md` 有一份规范。
- 规范定义接口、行为、不变量；代码是规范的实现。
- 规范与代码不一致时以规范为准。
- PR 不允许只改一侧。

## 理由

- **规范先行降低返工**：spec 阶段发现分歧的成本远低于实现阶段。
- **便于 Agent 协作**：Agent 读 spec 就能判断实现是否偏离契约，不需要逆向推理。
- **ADR + spec 双层记录**：spec 管"当前契约是什么"，ADR 管"为什么是这个契约"。
- 对比"代码即文档"：上一代的教训证明代码无法自我解释决策依据。

## 后果

- 正面：契约集中可审；Agent 能基于 spec 做有边界的自动开发。
- 正面：重构时 spec 作为锚点，避免误删行为。
- 负面：新增模块成本上升（多写一份 md）。对 trivial 工具函数可合并到相关模块 spec。
- 需要注意：spec 和代码同步是纪律问题；在 PR review 中硬性要求，并由 CI 检查 spec 覆盖率（后续 ADR 落实）。
